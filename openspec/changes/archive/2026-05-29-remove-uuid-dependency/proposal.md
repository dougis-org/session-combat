## GitHub Issues

- #240

## Why

- Problem statement: The project depends on the `uuid` npm package solely to call `v4()` (random UUID generation). Node.js 14.17+ ships `crypto.randomUUID()` natively — the same RFC 4122 v4 format, same CSPRNG source — making the package redundant.
- Why now: The `@types/uuid` devDependency is already mismatched (`uuid@9` but `@types/uuid@10`), which will cause type drift. Removing both cleans up the dependency graph before the mismatch causes a real issue.
- Business/user impact: Zero user-visible change. Pure maintenance win: one fewer dependency to audit, update, and bundle.

## Problem Space

- Current behavior: 3 source files and 2 E2E test helpers import `{ v4 as uuidv4 } from "uuid"` and call it to generate random IDs for monsters, spells, and test isolation tokens.
- Desired behavior: The same 5 files use `crypto.randomUUID()` from Node's built-in `crypto` module. The `uuid` and `@types/uuid` packages are removed from `package.json`.
- Constraints: Node 25.6.1 is the runtime — `crypto.randomUUID()` has been available since Node 14.17 (and as a global since Node 19), so no polyfill or version gate is needed.
- Assumptions: All UUID call sites are server-side or test-only (no browser bundle path that would require a polyfill). Confirmed: all 5 files are in `lib/`, `tests/e2e/`, — no client-side component.
- Edge cases considered: `isolation.ts` strips hyphens (`uuidv4().replace(/-/g, "")`). The native API produces the same hyphenated format, so `.replace(/-/g, "")` works identically.

## Scope

### In Scope

- Replace all 5 `import { v4 as uuidv4 } from "uuid"` with `import { randomUUID } from "crypto"`
- Replace all `uuidv4()` call sites with `randomUUID()`
- Remove `uuid` from `dependencies` in `package.json`
- Remove `@types/uuid` from `devDependencies` in `package.json`
- Run `npm install` to update `package-lock.json`

### Out of Scope

- Any other dependency audit or cleanup
- Changes to how IDs are stored, validated, or displayed
- Adding UUID validation logic

## What Changes

- `lib/api/spell-helpers.ts` — swap import + 1 call site
- `lib/import/transformMonster.ts` — swap import + 1 call site
- `lib/import/transformSpell.ts` — swap import + 1 call site
- `tests/e2e/helpers/actions.ts` — swap import + 1 call site
- `tests/e2e/helpers/isolation.ts` — swap import + 1 call site (hyphen-strip pattern unchanged)
- `package.json` — remove `uuid` and `@types/uuid`
- `package-lock.json` — regenerated by `npm install`

## Risks

- Risk: `crypto.randomUUID()` differs in some subtle edge case from `uuid` v4
  - Impact: IDs generated incorrectly
  - Mitigation: Both implement RFC 4122 §4.4 using the platform CSPRNG. Node's implementation is the reference. No behavior difference exists.

- Risk: A future contributor re-adds `uuid` not knowing it was intentionally removed
  - Impact: Dependency creep recurs
  - Mitigation: The `tasks.md` commit message will reference issue #240 so git blame explains the removal.

## Open Questions

No unresolved ambiguity. All call sites are confirmed, Node version is confirmed, and the native API is a drop-in replacement for `v4` usage.

## Non-Goals

- Auditing or replacing other third-party dependencies
- Switching to a different UUID version (v7, etc.)
- Adding runtime UUID validation
